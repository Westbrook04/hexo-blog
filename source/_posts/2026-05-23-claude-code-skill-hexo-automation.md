## 一、背景

我博客搭在 Hexo + Fluid 主题上，通过 GitHub Actions 自动部署到 GitHub Pages。之前每次写博客都要手动处理一堆步骤：

1. 手写 markdown 文件
2. 写 frontmatter（标题、日期、分类、标签）
3. 开代理（国内直连 GitHub 不稳定）
4. git add / commit / push
5. 关代理

虽然流程本身不复杂，但每次重复这些机械步骤还是挺烦的。

后来开始用 Claude Code，想着能不能让它帮我写博客。但一开始的 Skill 流程设计得太啰嗦了——每次发博客都要确认标题、确认分类、确认标签、确认全文，一步步点下来比我自己写还慢。

这篇文章记录一下怎么把 Skill 从"过度谨慎"改造成"一键完成"的。

## 二、改造思路

### 去掉所有审核步骤

最初的 Skill 有 5 个确认环节：

- 选模式 → 确认标题 → 选分类标签 → 确认全文预览 → 确认发布

改完后只剩一步：用户说 `/hexo-post`，我理解内容，直接出结果。

标题根据对话主题自动生成，分类标签参考博客已有文章风格自动匹配，正文按内容自然组织。用户不满意可以自己改 markdown，但不需要在流程里强制确认。

### 集成代理管理

国内访问 GitHub 不稳定是常态，FlClash（魔戒）平时关着节省资源。改造后的 Skill 在推送前自动检测代理是否运行，没运行就启动，等端口就绪再推送，推送完自动关掉。

关键信息：

- 可执行文件：`C:\Program Files\FlClash\FlClash.exe`
- 代理地址：`http://127.0.0.1:17890`
- 进程名：`clash-windows-amd64.exe`

启动检测用 `tasklist | grep`，就绪检测用 curl 请求 Google，超时设为 15 秒。

### 踩坑记录：TOC 目录不显示

第一次用新 Skill 发博客后，发现两个问题：

1. 首页检索不到（GitHub Actions 部署还没完成，等一会儿就好）
2. 文章目录只显示一个标题

第二个问题是 Hexo Fluid 主题的机制导致的：主题会从 frontmatter 的 `title` 字段渲染出一个 `h1` 标签，但如果 markdown 正文又以 `#` 开头，页面里就会出现**两个 `h1`**。tocbot（生成目录的 JS 库）在这种情况下行为异常，导致目录里只显示文章标题，各级小标题都消失。

修复很简单：正文里不再写 `#` 标题，直接从 `##` 开始分段。

## 三、最终效果

现在的流程是：

```
用户输入 /hexo-post
  → 我回顾对话内容
  → 自动生成标题 + 分类 + 标签
  → 写出 markdown 文件
  → 开代理 → git push → 关代理
  → 告诉用户搞定了
```

全程一次交互，不需要中间确认。

## 四、一点感受

做自动化工具时，"交互次数" 是一个重要的设计指标。

第一次的 Skill 设计把所有可自定义的参数都暴露给用户确认，看起来是"灵活"，实际上是把操作成本从开发者转嫁给了用户。去掉审核步骤后，妥协了一点点灵活性（用户不能在发布前调整分类标签），但换来了更快的操作体验。

如果用户对自动生成的分类标签不满意，直接改 markdown 文件再推一次就行——成本远低于每次都要走一遍确认流程。
