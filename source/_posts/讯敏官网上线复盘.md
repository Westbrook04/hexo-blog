# 讯敏官网上线复盘

## 一、项目背景

这次部署的项目是一个前后端分离的网站，目录结构如下：

- `cilent`：官网前台，基于 Vite 构建后的静态站点
- `admin`：后台管理端，基于 Vite 构建后的静态站点
- `service`：Node.js + Express 后端服务
- `service/database/xunmin_website.sql`：MySQL 初始化脚本

本次目标不是一次性把全部内容都公开上线，而是先完成官网前台上线，后台管理系统暂时不暴露公网。

## 二、部署目标

本次部署目标确定为：

- 域名：`www.xunminxinxis.com`
- 前台官网可公网访问
- 后端接口通过同域名 `/api` 转发
- 上传文件通过同域名 `/uploads` 转发
- 后台 `admin` 暂不开放公网

最终采用的部署方案是：

- 阿里云服务器
- 宝塔面板
- Nginx 提供静态站点与反向代理
- PM2 常驻运行 Node 服务
- MySQL 存储业务数据

## 三、最终部署架构

上线后的结构如下：

```text
用户访问 https://www.xunminxinxis.com
        |
        v
      Nginx
      |   \
      |    \-- /api/*      -> http://127.0.0.1:3000
      |    \-- /uploads/*  -> http://127.0.0.1:3000
      |
      \-- /               -> 前台静态文件 dist

Node service:
- 监听 127.0.0.1:3000
- 连接 MySQL
- 提供 /api 和 /uploads
```

这个方案的优点是前后端入口统一，前端不需要额外改线上接口地址，直接通过同域名访问即可。

## 四、这次实际遇到的问题

这次部署过程中，核心不是“不会部署”，而是连续遇到了几个很典型的环境问题。真正耗时的部分主要集中在环境联通、版本兼容、站点路由和账号权限上。

### 1. 不确定服务器部署方案

#### 现象

一开始项目目录里同时存在：

- `cilent`
- `admin`
- `service`

如果不先判断技术栈，很容易误以为是单体站点，或者直接把整个目录上传后乱配。

#### 分析

查看各目录的 `package.json`、`vite.config.js`、`service/src/server.js` 后确认：

- `cilent` 是前台静态站点
- `admin` 是后台静态站点
- `service` 是独立的 Node 接口服务
- `cilent` 和 `admin` 开发环境都通过 Vite 代理 `/api -> 127.0.0.1:3000`

说明线上不能只“传个前端”，而是必须：

- 前端打包后交给 Nginx
- Node 服务单独运行
- Nginx 负责把 `/api` 代理到 Node

#### 结论

必须先明确这不是纯静态站点，而是“静态前端 + Node API + MySQL”的标准分离部署。

### 2. 域名访问串到了另一个站点

#### 现象

部署过程中发现访问 `www.xunminxinxis.com` 时，会被带到另外一个站点 `pp.xunminxinxis.com`，表现为域名访问结果不符合预期。

#### 分析

这个问题本质上通常不是“代码错了”，而是站点绑定或 Nginx 配置错了。常见原因包括：

- 两个域名都解析到同一台服务器
- 宝塔里站点绑定域名有重叠
- 某个站点成了默认站点
- `server_name` 写错
- SSL 或重定向配置把 `www` 跳到了 `pp`

#### 解决思路

排查重点应放在：

1. 域名解析是否都指向同一台机器
2. 宝塔站点的域名绑定是否一一对应
3. Nginx 的 `server_name` 是否准确
4. 是否存在错误的 301/302 跳转

#### 经验

访问被跳到别的站点时，不要先怀疑代码，应先查：

- 域名解析
- 宝塔站点绑定
- Nginx `server_name`
- SSL 与强制跳转

### 3. 接口地址返回 502

#### 现象

访问：

```text
https://www.xunminxinxis.com/api/health
```

返回 `502 Bad Gateway`。

#### 初步判断

`502` 说明：

- Nginx 已经接到了请求
- 但 Nginx 转发给后端时失败了

这时候问题通常不在前端，而在下面几种情况里：

- Node 服务没启动
- Node 服务启动失败
- Node 服务监听端口不对
- Nginx 反代目标端口写错

#### 排查过程

通过 `pm2 logs xunmin-service --lines 100` 查看日志，发现服务根本没有成功启动，而是直接报了语法错误。

日志中关键报错：

```text
SyntaxError: Unexpected token '.'
```

#### 结论

`502` 只是表象，真正问题是后端没起来。

#### 经验

遇到 `502` 不要直接改 Nginx，先验证：

```bash
pm2 list
pm2 logs xunmin-service --lines 100
curl http://127.0.0.1:3000/api/health
```

先判断是不是 Node 本身没启动，再决定要不要动代理配置。

### 4. Node.js 版本过低导致服务启动失败

#### 现象

从 `pm2` 日志里看到：

```text
SyntaxError: Unexpected token '.'
```

进一步检查服务器 Node 版本：

```bash
node -v
```

结果是：

```text
v12.22.9
```

#### 分析

项目本身要求的 Node 版本是：

- 前后端包的 `engines` 指向 Node 20+
- `service` 至少要求 `node >=20.19.0`

而服务器当前跑的是 Node 12，无法解析项目里使用的现代 JavaScript 语法，所以服务启动时就直接崩掉了。

#### 解决过程

服务器中实际已经安装了 Node 20，只是默认环境变量仍然指向旧版 Node 12。

已安装的新版本路径：

```text
/www/server/nodejs/v20.20.2
```

通过临时切换 PATH 使用 Node 20：

```bash
export PATH=/www/server/nodejs/v20.20.2/bin:$PATH
node -v
npm -v
```

确认切换成功后，重新安装依赖并重启服务：

```bash
cd /www/wwwroot/xunmin-service
rm -rf node_modules
npm install
pm2 delete xunmin-service
pm2 start src/server.js --name xunmin-service
pm2 save
```

之后日志恢复正常：

```text
database connected
xunmin IT service listening on http://127.0.0.1:3000
GET /api/health 200
```

#### 根因

不是代码问题，而是服务器运行环境版本过低。

#### 经验

以后部署 Node 项目，先做这一步：

```bash
node -v
npm -v
```

必须先确认服务器 Node 版本满足项目要求，再安装依赖和启动服务。

### 5. phpMyAdmin 提示数据库账号无权限

#### 现象

在宝塔数据库相关操作中遇到报错：

```text
Access denied for user 'xunmin'@'localhost' (using password: YES)
```

#### 分析

这个报错不是网站代码报错，而是 MySQL 账号层面的认证失败。常见原因包括：

- 密码错误
- 用户未创建成功
- 用户权限未授予
- `localhost` 登录权限不正确

#### 处理思路

这个问题要和网站代码排查分开，不应该混在一起看。正确做法是单独验证数据库账号：

```bash
mysql -uxunmin -p
```

如果无法登录，就说明数据库用户本身有问题，需要：

- 重新创建数据库
- 重新创建数据库用户
- 重新设置密码
- 重新授权

#### 经验

数据库报 `Access denied` 时，先把问题收敛为“账号权限问题”，不要直接去怀疑 Node 代码或 Nginx 配置。

---

### 6. 不清楚阿里云和宝塔分别负责什么

#### 现象

部署过程中容易混淆：

- 哪些东西在阿里云控制台配置
- 哪些东西在宝塔里配置
- 哪些东西在项目代码里配置

#### 梳理后结论

阿里云主要负责：

- 域名解析
- 安全组放行端口

宝塔主要负责：

- 网站站点管理
- Nginx 配置
- PM2 服务管理
- MySQL 管理
- SSL 证书

项目代码主要负责：

- 前端打包产物
- 后端服务逻辑
- `.env` 环境变量

#### 本次实际需要放行的端口

- `80`：HTTP
- `443`：HTTPS
- `22`：SSH 管理

不建议公网开放：

- `3000`
- `3306`

#### 经验

以后排查问题时，先按层分清职责：

- 域名和公网访问问题查阿里云
- 站点和反代问题查宝塔/Nginx
- 服务运行问题查 PM2/Node
- 数据连接问题查 MySQL 和 `.env`

## 五、最终可用配置思路

### 1. 前台站点目录

网站根目录应直接指向前台打包后的静态文件目录，目录里应直接能看到：

- `index.html`
- `assets/`

而不是指向源码目录。

### 2. 后端 `.env`

后端最终应至少包含如下配置：

```env
PORT=3000
CORS_ORIGIN=https://www.xunminxinxis.com

DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=你的数据库用户名
DB_PASSWORD=你的数据库密码
DB_NAME=xunmin_website
```

### 3. Nginx 反向代理思路

站点核心逻辑如下：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}

location /api/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location /uploads/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

### 4. PM2 运行方式

服务启动方式：

```bash
pm2 start src/server.js --name xunmin-service
pm2 save
```

同时要确保 PM2 使用的是 Node 20 环境，而不是旧版 Node 12。

## 六、这次部署的核心经验

### 1. 先判断项目形态，再决定部署方式

不要一上来就上传文件。先弄清楚：

- 是纯静态站
- 还是前后端分离
- 是否依赖数据库
- 是否需要进程守护

### 2. 502 往往只是表象

`502` 很容易让人先去改 Nginx，但很多时候真正问题是：

- 后端没启动
- 后端崩了
- 端口没监听

### 3. 服务器环境版本必须先确认

Node 项目尤其要先确认：

- `node -v`
- `npm -v`

版本不对，后面所有步骤都可能白做。

### 4. 先做最小可用，再逐步开放

这次选择先上线官网前台，不急着开放后台公网，是正确策略。这样能先把主站跑通，再逐步开放管理端，降低复杂度。

### 5. 排查问题要按层切分

这次比较典型的几层分别是：

- 阿里云：域名解析、安全组
- 宝塔/Nginx：站点、反代、SSL
- Node/PM2：服务启动与常驻
- MySQL：数据库与账号权限
- 项目配置：`.env`、前端打包、目录路径

一旦分层清楚，问题定位会快很多。

## 七、下次部署前的检查清单

下次部署类似项目，建议先按下面顺序检查：

### 环境检查

- 服务器 Node 版本是否满足项目要求
- MySQL 是否已安装
- Nginx 是否已安装
- PM2 是否可用

### 域名与网络检查

- 域名是否已解析到服务器公网 IP
- 阿里云安全组是否已放行 `80/443`
- 服务器本机防火墙是否放行

### 项目部署检查

- 前台是否已执行 `npm run build`
- 网站目录是否指向正确的 `dist`
- 后端 `.env` 是否完整
- 数据库 SQL 是否已导入
- PM2 服务是否 `online`
- 本机 `curl http://127.0.0.1:3000/api/health` 是否正常
- 外部 `https://域名/api/health` 是否正常

## 八、总结

这次部署真正踩到的坑，不是业务代码本身，而是典型的服务器部署问题：

- 项目结构判断
- 域名与站点绑定
- Nginx 反向代理
- Node 版本兼容
- PM2 进程管理
- MySQL 账号权限

最后能顺利跑通，关键不在于“试很多次”，而在于把问题按层拆开，一层一层收敛。以后再遇到类似部署，只要先确认项目结构，再从环境、网络、服务、数据库四层逐步检查，效率会高很多。
