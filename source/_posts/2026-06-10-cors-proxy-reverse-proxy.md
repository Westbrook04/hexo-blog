今天在看一个官网后台项目的部署问题时，聊到了一个很常见但容易混乱的问题：本地开发时接口地址是 `127.0.0.1:3000`，上线后又变成官网域名，那是不是每次本地跑通、上线部署都要手动改代码？

答案是不应该。

这类问题的关键不是“把地址写在哪里更方便”，而是要先理解：浏览器为什么会跨域、代理为什么能绕开跨域、以及线上 nginx 到底在整个链路里做了什么。

## 问题的起点

项目里前端和后端是分开的：

```text
前端开发服务：127.0.0.1:5174
后端 Node 服务：127.0.0.1:3000
线上官网域名：www.xunminxinxis.com
```

本地开发时，如果前端代码直接请求：

```js
fetch('http://127.0.0.1:3000/api/products')
```

看起来很直接，但上线后这个地址就不能用了。因为用户浏览器里的 `127.0.0.1` 指的是用户自己的电脑，不是服务器。

反过来，如果代码里直接写：

```js
fetch('https://www.xunminxinxis.com/api/products')
```

本地开发又会直接请求线上接口，不方便调试本地后端，也容易把测试数据打到线上。

所以真正的问题是：不应该靠改源码来切换本地和线上环境。

## 什么是跨域

浏览器判断两个地址是不是同源，主要看三件事：

```text
协议 + 域名 + 端口
```

只要其中一个不同，就是跨域。

例如：

```text
http://127.0.0.1:5174
http://127.0.0.1:3000
```

它们协议和域名一样，但端口不同，所以在浏览器看来仍然是跨域。

线上也一样：

```text
https://www.xunminxinxis.com
http://127.0.0.1:3000
```

协议、域名、端口都不同，当然也是跨域。

浏览器限制跨域，是为了防止恶意网页随便请求别的网站接口。否则一个陌生网站就可能偷偷请求你的后台、邮箱、银行系统接口。

## CORS 解决的是什么

CORS 是后端告诉浏览器：

```text
这个来源可以访问我
```

后端响应里通常会带类似这样的头：

```http
Access-Control-Allow-Origin: http://127.0.0.1:5174
```

开发阶段也有人会写：

```http
Access-Control-Allow-Origin: *
```

但这只是让浏览器放行跨域响应，并不改变请求本身是跨域这件事。

更重要的是：CORS 是浏览器的安全策略。不是后端完全收不到请求，而是浏览器不允许前端 JavaScript 拿到不被允许的响应。

## 本地开发代理的原理

更推荐的方式是让前端代码永远请求相对路径：

```js
fetch('/api/products')
```

本地开发时，浏览器打开的是：

```text
http://127.0.0.1:5174
```

所以浏览器实际请求的是：

```text
http://127.0.0.1:5174/api/products
```

这和页面本身同源，不会触发跨域。

然后 Vite 开发服务器在背后做一层代理：

```text
浏览器
  -> http://127.0.0.1:5174/api/products
  -> Vite dev server
  -> http://127.0.0.1:3000/api/products
  -> Node 后端
```

浏览器看到的是同源请求，真正转发到 `3000` 端口的是 Vite。这个转发发生在开发服务器和后端之间，不是浏览器直接跨域访问后端。

这就是为什么本地的 `vite.config.js` 里通常会有：

```js
server: {
  proxy: {
    '/api': 'http://127.0.0.1:3000'
  }
}
```

## 线上 nginx 做的是同一件事

上线后，Vite 开发服务器不存在了，前端已经被构建成静态文件。

用户访问：

```text
https://www.xunminxinxis.com
```

前端代码依然请求：

```js
fetch('/api/products')
```

浏览器实际请求：

```text
https://www.xunminxinxis.com/api/products
```

这和页面仍然是同源的。

然后 nginx 根据路径规则转发：

```text
浏览器
  -> https://www.xunminxinxis.com/api/products
  -> nginx
  -> http://127.0.0.1:3000/api/products
  -> Node 后端
```

这里的 `127.0.0.1:3000` 是服务器自己访问自己，不是用户电脑的 `127.0.0.1`。

也就是说，本地和线上本质是同一个思路：

```text
本地：Vite 负责代理
线上：nginx 负责代理
前端代码：永远请求 /api
```

## 什么是反向代理

代理分两种：正向代理和反向代理。

正向代理是代理客户端。比如用户通过代理服务器访问外网：

```text
用户 -> 代理服务器 -> 目标网站
```

目标网站看到的是代理服务器，不一定知道真实用户是谁。

反向代理是代理服务器端。用户访问的是 nginx：

```text
用户 -> nginx -> 后端服务
```

用户不知道 nginx 后面真正处理请求的是哪个服务。它可能是：

```text
127.0.0.1:3000
127.0.0.1:3001
另一台服务器
多个后端实例
```

对用户来说，他只知道：

```text
https://www.xunminxinxis.com
```

所以可以简单记：

```text
正向代理：帮用户去访问别人
反向代理：帮服务器接待用户
```

官网项目里的 nginx 就是反向代理。

## 最推荐的配置方式

前端代码不要写死本地地址，也不要写死线上域名，而是统一写成：

```js
const API_BASE_URL = (import.meta.env.VITE_API_BASE_URL || '').replace(/\/+$/, '')
const API_PREFIX = `${API_BASE_URL}/api`
```

如果 `VITE_API_BASE_URL` 为空，请求就是：

```text
/api/products
```

本地由 Vite 代理：

```text
/api -> http://127.0.0.1:3000
```

线上由 nginx 代理：

```text
/api -> http://127.0.0.1:3000
```

这样本地开发和线上部署都不需要改源码。

只有当前端和后端部署在不同域名时，才需要显式配置：

```env
VITE_API_BASE_URL=https://api.example.com
```

如果前端、后台、接口都在同一个官网域名下，优先使用同域 `/api`，更简单，也更少遇到跨域问题。

## 这次的收获

以前容易把问题理解成“本地地址和线上地址怎么同步”，但更准确的理解应该是：

```text
源码不负责环境切换
环境配置负责环境差异
代理层负责请求转发
```

本地跑通靠 Vite proxy，线上访问靠 nginx reverse proxy。前端代码只请求 `/api`，不用知道后端真实运行在哪个端口。

这样项目才不会出现“本地改成 127，上线又改成域名”的来回切换，也不会因为一次手动修改忘记还原，把本地配置误带到线上。
